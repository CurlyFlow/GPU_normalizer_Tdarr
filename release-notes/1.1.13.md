## TLDR

- Speeds up the default `gpuSourcePort` path again for long generated-stereo fallback jobs while preserving exact decoded parity against Tdarr CPU `Normalize Audio`.
- Full required matrix passed exact decoded parity for both streams: original 5.1 and generated normalized 2-channel fallback.
- 60min split: original 5.1 is `2.254x` faster (`1027.3s` SRC_CPU vs `455.8s` GPU), generated 2ch is `0.673x` by itself (`291.2s` SRC_CPU vs `432.5s` GPU), and the combined job is `1.476x` faster (`1318.5s` SRC_CPU vs `893.1s` GPU).
- Compared with `1.1.12`, the combined long-media GPU job is `1.265x` faster on 10min, `1.244x` faster on 30min, and `1.231x` faster on 60min.

## Performance

`1.1.13` was compared against Tdarr's legacy CPU-only `Normalize Audio` output. The benchmark source is a 5.1 file without an existing stereo track, so the accepted GPU job outputs both the normalized original 5.1 stream and a generated normalized 2-channel fallback. Both decoded output streams were compared against CPU output and passed with `energy_similarity=1.000000000` and `rms_diff=0`.

`SRC_CPU` means the Tdarr CPU `Normalize Audio` reference/cache. For generated fallback, the harness creates the matching 2-channel source and runs CPU `Normalize Audio` on it. `OUR_CPU` means CPU-side work inside the GPU plugin run.

`Speed vs SRC_CPU` above `1.0x` means the GPU plugin was faster than CPU for that case. Values below `1.0x` mean the GPU plugin was slower.

## Changes

- Promotes the parity-clean exact-source generated-stereo fast path to defaults.
- Lets generated-stereo exact-source chunks skip safe feedback before the threshold lock when the precomputed exact source sums prove it is safe.
- Uses the source-specific 2-channel unsafe skip kernel by default for generated-stereo exact-source work.
- Coalesces exact-source stats chunks to two windows by default, reducing stats/prefix overhead while preserving exact decoded parity.
- Keeps diagnostic source shortcut math paths default-off because they can be faster but failed long 60min decoded parity.

## Caveats

- Only `192k` AAC output and the tested Tdarr CPU `Normalize Audio` contract are covered by this release matrix.
- Generated 2ch fallback is still slower than CPU by itself; the combined long job wins because original 5.1 normalization and overlapped CPU/GPU work dominate over long durations.
- Short clips can still be slower or near-neutral because startup, stream setup, and FFmpeg overhead dominate.
- Release codec smoke passed exact decoded parity for AAC stereo, MP3 stereo, AC3 5.1, E-AC-3 5.1, DTS 5.1, and multi-audio.
- `SRC_CPU` is a waveform oracle for CPU `Normalize Audio` behavior. Plugin-only orchestration, such as stream reordering or generated-track placement, is structural plugin behavior and is not represented as something CPU `Normalize Audio` can perform directly.
