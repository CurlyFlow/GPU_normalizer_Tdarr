## TLDR

- `1.2.4` is a feature and maintainability release over `1.2.3`, not a speed-target release.
- The full final installed-plugin matrix passed decoded parity: `30s`, `60s`, `10min`, `30min`, and `60min`.
- Long combined jobs remain faster than Tdarr CPU `Normalize Audio`: `1.620x` on `10min`, `1.736x` on `30min`, and `1.619x` on `60min`.
- New language-scope safety: when selected languages do not exist, `Remove Other Languages=true` now keeps undetected audio streams unchanged if present; if no desired or undetected audio would remain, the job fails instead of producing a no-audio output.
- Codec smoke passed decoded parity for AAC stereo, MP3 stereo, AC3 5.1, E-AC-3 5.1, DTS 5.1, multi-audio language-scope behavior, and the new no-language-match hard-fail behavior.

## Performance

`1.2.4` was compared against Tdarr's legacy CPU-only `Normalize Audio` output. The benchmark source is a 5.1 file without an existing stereo track, so the accepted GPU job outputs both the normalized original 5.1 stream and a generated normalized 2-channel fallback. Both decoded output streams were compared against CPU output and passed with `energy_similarity=1.000000000`.

`SRC_CPU` means the Tdarr CPU `Normalize Audio` reference/cache. For generated fallback, the harness creates the matching 2-channel source and runs CPU `Normalize Audio` on it. `OUR_CPU` means CPU-side work inside the GPU plugin run.

`Speed vs SRC_CPU` above `1.0x` means the GPU plugin was faster than CPU for that case. Values below `1.0x` mean the GPU plugin was slower.

### 60min

| Workload | SRC_CPU reference | GPU Normalize Audio `1.2.4` | Speed vs SRC_CPU | Decoded audio vs SRC_CPU output |
| --- | ---: | ---: | ---: | --- |
| Original 5.1 | `1027.3s` | `814.2s` | `1.262x` | 5.1 passed, `energy_similarity=1.000000000`. |
| Generated 2ch | `291.2s` | `814.2s` | `0.358x` | +2ch passed, `energy_similarity=1.000000000`. |
| Combined | `1318.5s` | `814.3s` | `1.619x` | Both streams passed. |

## Changes

- Adds `Fallback To Undetected Audio`, default `true`. With `Normalize ONLY Languages` set and `Remove Other Languages=true`, if no selected language exists but undetected audio exists, undetected audio is copied unchanged instead of being removed.
- Changes no-match behavior when removal is on: if the output would contain no audio streams, the plugin now fails with a clear setup error instead of producing a no-audio file.
- Keeps the `1.2.3` defaults for `Normalize ONLY Languages=eng,en`, `Remove Other Languages=true`, generated first-language 2-channel fallback, streaming `gpuSourcePort`, and decoded parity priority.
- Promotes the hard `review-fix` architecture cleanup: context projection ownership, canonical source-audio action planning, stream plan state/backend separation, explicit split-stats task records, split-stats process graph rendering, safer paired-apply gain gating, sectioned runtime config, and table-driven runtime profiling.
- Improves release tooling: packaging now requires release-note-quality text, and the compare harness preserves failed GPU diagnostics under stable failure paths before GPU artifact pruning.

## Codec Smoke

Release codec smoke passed decoded parity for AAC stereo, MP3 stereo, AC3 5.1, E-AC-3 5.1, DTS 5.1, and multi-audio.

The multi-audio smoke also verifies language-scope behavior:

| Scenario | Result |
| --- | --- |
| blank/all languages | Passed; all audio kept/normalized. |
| omitted default | Passed; default English selection removed other languages. |
| selected languages only | Passed; selected languages kept, others removed. |
| selected language with `Remove Other Languages=false` | Passed; the selected language was normalized and other languages copied. |
| no language match | Passed; plugin fails safely when no desired or undetected audio would remain. |

## Caveats

- Only `192k` AAC output and the tested Tdarr CPU `Normalize Audio` contract are covered by this release validation.
- Generated 2ch fallback is still slower than CPU by itself.
- `1.2.4` is not a speed-win release over `1.2.3`; it is a feature/refactor release that keeps the long combined workload faster than CPU.
- Short clips can be slower because startup, stream setup, decode, and encode overhead dominate.
- Language filtering uses source metadata. Wrong or missing language tags cannot be corrected automatically.
