## TLDR

- Speeds up the limiter-active `gpuSourcePort` path while keeping decoded output matched to Tdarr's CPU-only `Normalize Audio` on the required matrix.
- Keeps streaming/two-pass behavior so long media does not create huge raw PCM bridge files.
- Still slower than CPU normalize on long limiter-active media; this is an incremental GPU speed release, not a faster-than-CPU claim.

## Performance

`1.1.3` was compared against Tdarr's legacy CPU-only `Normalize Audio` plugin output. All accepted cases matched decoded audio with `energy_similarity=1.000000000`.

| Case | Legacy CPU `Normalize Audio` | GPU Normalize Audio `1.1.3` | Speed vs CPU | Decoded audio vs CPU output |
| --- | ---: | ---: | ---: | --- |
| 12s | `4.2s` | `2.3s` | `1.852x` | Passed, `energy_similarity=1.000000000`. |
| 60s | `16.7s` | `55.7s` | `0.300x` | Passed, `energy_similarity=1.000000000`. |
| 30min | `544.0s` | `1695.0s` | `0.321x` | Passed, `energy_similarity=1.000000000`. |

Compared with the previous `1.1.2` GPU release on the required 30min case, wall time improved from `2488.1s` to `1695.0s`, and exact apply improved from `1992.9s` to `1168.9s`.

## Changes

- Enabled sparse exact-limiter scheduling for limiter-active 5.1 content.
- Prefills safe output windows in parallel, runs exact serial true-peak limiter on risk-expanded unsafe regions, and keeps exact output feedback for deterministic CPU-normalizer parity.
- Uses a conservative default unsafe-region expansion of `10` windows; a raw unsafe-only diagnostic was faster on short clips but failed the 30min parity case and is not the release default.
- Optimized the 6-channel feedback-only path and avoided unnecessary limiter-buffer fill work in safe regions.
- Keeps diagnostic environment overrides for future performance work.
