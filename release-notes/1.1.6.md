## TLDR

- Speeds up limiter-heavy `gpuSourcePort` jobs by replacing most exact safe feedback work with a segmented CUDA safe-feedback path for 5.1 audio.
- Keeps decoded output matched to Tdarr's CPU-only `Normalize Audio` on the required 30s, 60s, and 30min matrix.
- This is a GPU speed release versus `1.1.5`, but it is still slower than CPU-only `Normalize Audio` on the measured matrix.

## Performance

`1.1.6` was compared against Tdarr's legacy CPU-only `Normalize Audio` plugin output. All accepted cases matched decoded audio with `energy_similarity=1.000000000`.

| Case | Legacy CPU `Normalize Audio` | GPU Normalize Audio `1.1.6` | Speed vs CPU | Decoded audio vs CPU output |
| --- | ---: | ---: | ---: | --- |
| 30s | `10.1s` | `20.5s` | `0.493x` | Passed, `energy_similarity=1.000000000`. |
| 60s | `16.7s` | `36.5s` | `0.458x` | Passed, `energy_similarity=1.000000000`. |
| 30min | `544.0s` | `964.1s` | `0.564x` | Passed, `energy_similarity=1.000000000`. |

Compared with `1.1.5`, the 30min GPU time improved from `1513.4s` to `964.1s`, a `36.3%` wall-time reduction. Exact apply time on the same case improved from about `1158.5s` to about `615.0s`; exact stats remained around `114.8s`.

Do not read the `Speed vs CPU` column as a faster-than-CPU claim. Values below `1.0x` mean GPU is still slower than CPU-only `Normalize Audio` for that case.

## Changes

- Adds default-on segmented exact safe feedback for 5.1 f64 safe chunks.
- Keeps unsafe limiter regions on the existing exact limiter path to preserve decoded parity.
- Adds profiling for segmented safe-feedback chunks, windows, fallbacks, and apply time.
- Keeps the normal streaming/two-pass `gpuSourcePort` behavior so long media does not create huge raw PCM bridge files.
