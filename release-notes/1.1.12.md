## TLDR

- Speeds up the default `gpuSourcePort` path for long generated-stereo fallback jobs while preserving exact decoded parity against the corrected Tdarr CPU `Normalize Audio` contract.
- Full required matrix passed exact decoded parity for both streams: original 5.1 and generated normalized 2-channel fallback.
- 60min split: original 5.1 is `1.747x` faster (`1027.3s` SRC_CPU vs `588.1s` GPU), generated 2ch is `0.574x` by itself (`291.2s` SRC_CPU vs `507.3s` GPU), and the combined job is `1.199x` faster (`1318.5s` SRC_CPU vs `1099.6s` GPU).
- Compared with `1.1.11`, the combined long-media GPU job is `1.105x` faster on 10min, `1.117x` faster on 30min, and `1.125x` faster on 60min.

## Performance

`1.1.12` was compared against Tdarr's legacy CPU-only `Normalize Audio` output. The benchmark source is a 5.1 file without an existing stereo track, so the accepted GPU job outputs both the normalized original 5.1 stream and a generated normalized 2-channel fallback. Both decoded output streams were compared against CPU output and passed with `energy_similarity=1.000000000` and `rms_diff=0`.

`SRC_CPU` means the Tdarr CPU `Normalize Audio` reference/cache. For generated fallback, the harness creates the matching 2-channel source and runs CPU `Normalize Audio` on it. `OUR_CPU` means CPU-side work inside the GPU plugin run.

`Speed vs SRC_CPU` above `1.0x` means the GPU plugin was faster than CPU for that case. Values below `1.0x` mean the GPU plugin was slower.

## Changes

- Fuses generated-stereo source-exact precompute into the existing channel-stats decode pass, avoiding a separate FFmpeg decode while preserving the same exact source-energy/source-sums kernels.
- Increases generated-stereo source precompute batching and adds a memory cap so long inputs use fewer, better-occupied launches without unbounded staging.
- Reduces exact risk expansion to the minimum that passed the required matrix, cutting conservative limiter feedback work on long files.
- Starts the next stream's OUR_CPU loudnorm first pass earlier so original-stream CPU work overlaps the generated fallback GPU path.
- Enables bounded exact apply-decode prefetch with exact apply chunk sizing and removes avoidable byte-copy staging in hot streaming paths.
- Makes non-exact streaming apply fill pipe reads robustly and zero-pad only true EOF tail to the first-pass length, matching CPU `Normalize Audio` tail behavior on linear codec-smoke paths.
- Keeps exact apply-decode prefetch out of non-exact linear streaming, avoiding pipe pre-read shifts on AC3 and multi-audio codec smoke cases.
- Clarifies the compare harness around `SRC_CPU` versus `OUR_CPU`, with missing `SRC_CPU` caches auto-created and `--refresh-cpu` reserved for OUR_CPU/GPU reruns.

## Caveats

- Only `192k` AAC output and the tested Tdarr CPU `Normalize Audio` contract are covered by this release matrix.
- Generated 2ch fallback is still slower than CPU by itself; the combined long job wins because original 5.1 normalization and overlapped CPU/GPU work dominate over long durations.
- Short clips can be slower than CPU because startup, stream setup, and FFmpeg overhead dominate.
- Release codec smoke passed exact decoded parity for AAC stereo, MP3 stereo, AC3 5.1, E-AC-3 5.1, DTS 5.1, and multi-audio.
- `SRC_CPU` is a waveform oracle for CPU `Normalize Audio` behavior. Plugin-only orchestration, such as stream reordering or generated-track placement, is structural plugin behavior and is not represented as something CPU `Normalize Audio` can perform directly.
