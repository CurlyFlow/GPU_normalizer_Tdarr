## TLDR

- Exact-parity GPU release: short benchmark outputs matched Tdarr's CPU-only `Normalize Audio` decoded output.
- The exact CUDA apply path is faster than the prior `1.1` candidate line, but measured GPU runtime is still slower than CPU.
- Long-media performance is not accepted as faster than CPU in this release.

## Performance

`1.1` keeps the CPU-output-matching `gpuSourcePort` behavior from `1.0` and improves the exact 5.1 CUDA apply path used by the short benchmark cases. It remains correctness-first: measured GPU output matched Tdarr's CPU-only `Normalize Audio` decoded output exactly on the short matrix, but GPU runtime was still slower than CPU.

| Case | Legacy CPU `Normalize Audio` | GPU Normalize Audio `1.1` | Speed vs CPU | Decoded audio vs CPU output |
| --- | ---: | ---: | ---: | --- |
| 12s 5.1 smoke | `3.8s` | `8.3s` | `0.455x` | Passed: `energy_similarity=1`, `rms_diff=0`, `equal_fraction=1`. |
| 30s public movie clip | `9.7s` | `21.1s` | `0.457x` | Passed: `energy_similarity=1`, `rms_diff=0`, `equal_fraction=1`. |

The main speedup in this release is inside the exact CUDA apply path. The 30s apply stage improved from the prior short candidate's `24.5s` to `17.4s`, while preserving `ffmpeg_limiter=1` and `ffmpeg_final_flush=1` behavior.

Long-media performance is not accepted as faster than CPU in this release. Use `1.1` for the latest exact-parity GPU implementation and roll back to `1.0` if your own workflow prefers the previous stable line.

## Changes

- Optimized the exact 5.1 CUDA apply hot path with direct limiter-buffer writes, localized output-feedback IIR state, and lower-overhead gain interpolation.
- Kept `gpuSourcePort` as the default planner mode and preserved the versioned release folder layout.
- Kept compact benchmark output and optional debug/profiling hooks in the maintainer harness; these are not user-facing plugin behavior changes.
