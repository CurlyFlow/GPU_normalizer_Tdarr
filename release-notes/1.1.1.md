## TLDR

- Keeps the `1.1` CPU-normalizer-matching audio path unchanged.
- Adds guarded, configurable GPU normalize concurrency using slot lock directories with heartbeat and stale cleanup.
- Defaults to one concurrent GPU normalize job per lock base; set `Max Concurrent Jobs` in the plugin to change it.

## Performance

`1.1.1` does not change the loudnorm planner, CUDA renderer, or AAC encode path from `1.1`. Audio parity and runtime expectations are therefore the same as `1.1`: measured short benchmark outputs matched Tdarr's CPU-only `Normalize Audio` decoded output, but GPU runtime remained slower than CPU.

| Case | Legacy CPU `Normalize Audio` | GPU Normalize Audio `1.1.1` | Speed vs CPU | Decoded audio vs CPU output |
| --- | ---: | ---: | ---: | --- |
| 12s 5.1 smoke | `3.8s` | `8.3s` | `0.455x` | Same audio path as `1.1`; `1.1` passed with `energy_similarity=1`, `rms_diff=0`, `equal_fraction=1`. |
| 30s public movie clip | `9.7s` | `21.1s` | `0.457x` | Same audio path as `1.1`; `1.1` passed with `energy_similarity=1`, `rms_diff=0`, `equal_fraction=1`. |

The concurrency guard limits how many plugin instances run at once; it is a scheduling/resource-safety feature, not a per-file speedup. Long-media performance remains not accepted as faster than CPU.

## Changes

- Added plugin inputs `Max Concurrent Jobs` and `Lock File`.
- `Max Concurrent Jobs` defaults to `1`; `0` disables the guard.
- Added slot lock directories named `${Lock File}.slotN.lockdir` with `holder` and `heartbeat` files.
- Added stale lock cleanup and active-process preservation based on the prod `opxGuardedNormalizeAudio` pattern.
- Kept `1.1` as an immutable rollback snapshot and released the guard as `1.1.1`.
