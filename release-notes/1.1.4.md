## TLDR

- Speeds up limiter-active `gpuSourcePort` again while keeping decoded output matched to Tdarr's CPU-only `Normalize Audio` on the required matrix.
- The main win is faster exact f64 stats for 5.1 limiter-active media; the required 30min case is about `11%` faster than `1.1.3`.
- Still slower than CPU normalize on long limiter-active media; this is another incremental GPU speed release, not a faster-than-CPU claim.

## Performance

`1.1.4` was compared against Tdarr's legacy CPU-only `Normalize Audio` plugin output. All accepted cases matched decoded audio with `energy_similarity=1.000000000`.

| Case | Legacy CPU `Normalize Audio` | GPU Normalize Audio `1.1.4` | Speed vs CPU | Decoded audio vs CPU output |
| --- | ---: | ---: | ---: | --- |
| 12s | `4.2s` | `2.3s` | `1.852x` | Passed, `energy_similarity=1.000000000`. |
| 60s | `16.7s` | `49.9s` | `0.335x` | Passed, `energy_similarity=1.000000000`. |
| 30min | `544.0s` | `1507.6s` | `0.361x` | Passed, `energy_similarity=1.000000000`. |

Compared with the previous `1.1.3` GPU release on the required 30min case, wall time improved from `1695.0s` to `1507.6s`, and exact f64 stats improved from `285.1s` to `114.8s`.

## Changes

- Optimizes exact f64 stats for aligned 6-channel limiter-active audio while preserving decoded parity.
- Moves raw peak detection into a parallel CUDA prepass so serial channel stats no longer scan all channels for peaks.
- Uses a default unsafe-region expansion of `5` windows for sparse limiter scheduling. A raw unsafe-only diagnostic was faster on short clips but failed the 30min parity case and is not the release default.
- Keeps streaming/two-pass behavior so long media does not create huge raw PCM bridge files.
- Keeps diagnostic environment overrides for future performance work.
